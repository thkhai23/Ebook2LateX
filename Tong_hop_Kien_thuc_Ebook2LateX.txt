TỔNG HỢP KIẾN THỨC DỰ ÁN EBOOK2LATEX (DỰA TRÊN BLOG PHÂN TÍCH)
==========================================================================

1. TÓM TẮT NỘI DUNG CẦN NHỚ
--------------------------------------------------------------------------
- Mục tiêu dự án: Xây dựng hệ thống chuyển đổi công thức toán học từ tài liệu PDF 
  sang định dạng LaTeX chuẩn quốc tế.
- Kiến trúc 4 thành phần cốt lõi:
    1. Frontend: React (Giao diện người dùng)
    2. Backend: FastAPI (Xử lý logic và API)
    3. Database: PostgreSQL (Lưu trữ lâu dài)
    4. AI Pipeline: Xử lý OCR/OCR-Math (Trích xuất công thức)
- Luồng dữ liệu chính: 
    Upload PDF -> AI xử lý -> Hiện kết quả lên UI -> Người dùng chỉnh sửa (MathLive) 
    -> Lưu vào Database.

2. TRẢ LỜI CÁC CÂU HỎI TRONG BLOG (BÀI TẬP 1A - 1E)
--------------------------------------------------------------------------
Câu 1a. PDF là gì? Tại sao được sử dụng nhiều?
- Trả lời: PDF (Portable Document Format) là định dạng tài liệu di động. Nó được 
  sử dụng rộng rãi vì giữ nguyên định dạng, font chữ, hình ảnh trên mọi thiết bị 
  và hệ điều hành khác nhau.

Câu 1b. LateX là gì? Dùng để làm gì?
- Trả lời: LaTeX là một hệ thống soạn thảo văn bản chất lượng cao. Nó được dùng 
  để soạn thảo các tài liệu khoa học, kỹ thuật và toán học nhờ khả năng trình bày 
  công thức toán cực kỳ chuyên nghiệp và chính xác.

Câu 1c. Mathlive symbol là gì? Tại sao phải chuyển đổi qua lại?
- Trả lời: MathLive là thư viện giao diện cho phép nhập công thức toán học trực quan 
  (gõ đến đâu hiện ký hiệu đến đó). Cần chuyển đổi vì:
    - PDF -> LaTeX: Để máy tính có thể hiểu và lưu trữ dưới dạng mã.
    - LaTeX -> Mathlive: Để người dùng (không biết mã LaTeX) vẫn có thể chỉnh sửa 
      công thức một cách dễ dàng qua giao diện đồ họa.

Câu 1d. Nên làm Desktop app hay Web app?
- Trả lời: Tùy mục đích, nhưng Web App (như dự án này) ưu việt hơn về khả năng 
  tiếp cận (chạy trên trình duyệt, không cần cài đặt), dễ dàng cập nhật và chia sẻ.

Câu 1e. So sánh CSR (Client-side) và SSR (Server-side)?
- Trả lời:
    - CSR: Tải trang nhanh sau lần đầu, tương tác mượt mà (React). Nhược điểm là SEO kém hơn.
    - SSR: Tốt cho SEO vì nội dung được tạo từ Server. Nhược điểm là tải trang chậm 
      hơn khi có nhiều tương tác phức tạp.

3. BỘ CÂU HỎI PHẢN BIỆN GIẢ ĐỊNH (TỪ GIẢNG VIÊN)
--------------------------------------------------------------------------
Câu 1: Tại sao em sử dụng PostgreSQL mà không phải Excel hay file Text để lưu dữ liệu?
- Trả lời: PostgreSQL là hệ quản trị CSDL quan hệ mạnh mẽ, hỗ trợ lưu trữ dữ liệu 
  lớn, đảm bảo tính toàn vẹn (integrity), hỗ trợ kiểu dữ liệu JSONB (để lưu Log) 
  và có khả năng mở rộng tốt hơn nhiều so với file text đơn giản.

Câu 2: Làm sao em xử lý được việc người dùng sửa công thức sai định dạng LaTeX?
- Trả lời: Thư viện MathLive tích hợp sẵn bộ kiểm tra cú pháp. Ngoài ra, ở Backend 
  FastAPI, chúng em có thể bổ sung thêm các bước validation để đảm bảo chỉ những 
  chuỗi LaTeX hợp lệ mới được lưu vào DB.

Câu 3: Vai trò của SQLAlchemy và Alembic trong dự án này là gì?
- Trả lời: SQLAlchemy đóng vai trò là ORM (ánh xạ code Python sang SQL) giúp thao tác 
  với DB dễ dàng. Alembic dùng để quản lý "phiên bản" của CSDL (Migrations), giúp 
  đồng bộ cấu trúc bảng giữa các máy tính của các thành viên trong nhóm.

Câu 4: Nếu PDF có hàng nghìn trang, hệ thống của em xử lý thế nào để không bị "treo"?
- Trả lời: Chúng em sẽ sử dụng hàng đợi (Task Queue như Celery) để xử lý ngầm (Background). 
  Người dùng upload xong có thể làm việc khác, khi nào AI xử lý xong sẽ thông báo lại.

Câu 5: Tính năng "Mua giày" có liên quan gì đến dự án Ebook2LateX không?
- Trả lời: Đó là một Endpoint thử nghiệm (Test Endpoint) để chứng minh khả năng xử lý 
  Path Parameters của FastAPI, giúp kiểm tra hệ thống có nhận và phản hồi đúng tham số 
  từ URL hay không trước khi triển khai các logic phức tạp hơn.

1. CẤU TRÚC VÀ CÔNG NGHỆ CHI TIẾT
--------------------------------------------------------------------------

- Trình soạn thảo (Frontend): Kết hợp React State và MathLive để tạo trình 
  soạn thảo hai chiều (sửa LaTeX hiện ký hiệu và ngược lại).
- UI Framework: Tailwind CSS giúp dàn trang nhanh và đẹp.

- Quản lý mã nguồn: Git

- Framework Frontend: React
- Framework Backend: FastAPI
- Trích xuất ảnh PDF: PyMuPDF (fitz)
- OCR Toán học: pix2tex

Câu 2.1 Thư viện trích xuất ảnh từ PDF là gì?
- Trả lời: B. PyMuPDF (fitz).

Câu 2.2 Tại sao sử dụng React State để quản lý dữ liệu soạn thảo?

3. BỘ CÂU HỎI PHẢN BIỆN GIẢ ĐỊNH (PHẦN HỆ THỐNG)
--------------------------------------------------------------------------
Câu 1: Tại sao em phải dùng Docker mà không cài trực tiếp thư viện vào máy?
- Trả lời: Các thư viện AI như pix2tex rất nặng và phụ thuộc vào phiên bản Python, 
  CUDA... Docker giúp đóng gói tất cả vào một "container" chuẩn, giúp việc triển 
  khai lên Server hoặc máy bạn khác không bị lỗi xung đột môi trường.

Câu 2: Cơ chế đồng bộ giữa MathLive và Textarea hoạt động như thế nào?
- Trả lời: Em sử dụng một State trung gian (latexContent). Khi người dùng gõ vào 
  Textarea, nó cập nhật State -> MathLive nhận State mới và render. Khi người 
  dùng sửa trên MathLive, nó trả về giá trị LaTeX mới -> cập nhật State -> 
  Textarea nhận giá trị mới.

Câu 3: FastAPI giúp ích gì trong việc chạy các Model AI nặng?
- Trả lời: FastAPI hỗ trợ "Worker processes" và xử lý bất đồng bộ. Điều này cho 
  phép Backend tiếp tục nhận các yêu cầu khác trong khi Model AI đang tính toán 
  một công thức phức tạp, tránh việc hệ thống bị nghẽn (bottleneck).

Câu 4: Làm sao em quản lý được hàng trăm file PDF người dùng upload lên?
- Trả lời: Chúng em lưu file thực tế trong thư mục "uploads" và chỉ lưu "đường dẫn" 
  (path) vào bảng Documents trong PostgreSQL. Cách này giúp Database nhẹ hơn 
  và truy xuất file nhanh hơn.

- B8 (Push): Đẩy mã nguồn lên Github lần đầu bằng: `git push -u origin main`.

PHẦN 6: PHÂN TÍCH VÀ THIẾT KẾ CÁC BẢNG (DỰA TRÊN BLOG BÀI 6)
==========================================================================

1. HỆ THỐNG 4 BẢNG CƠ SỞ DỮ LIỆU
--------------------------------------------------------------------------
- Bảng Documents: Lưu thông tin PDF (FileName, FilePath, UploadDate, Status).
- Bảng FormulaEntries: Lưu kết quả OCR (LatexContent, ImagePath, OrderIndex). 
  Liên kết với Documents qua DocumentID (Khóa ngoại).
- Bảng Users: Quản lý người dùng (Username, PasswordHash, Role).
- Bảng Logs: Theo dõi AI (ProcessingTime, ConfidenceScore, ErrorMessage).

2. CHI TIẾT CÁC TRƯỜNG DỮ LIỆU QUAN TRỌNG
--------------------------------------------------------------------------
- Primary Key (Khóa chính): Định danh duy nhất cho mỗi dòng trong bảng.
- Foreign Key (Khóa ngoại): Tạo mối quan hệ giữa các bảng (ví dụ: Công thức 
  thuộc về Tài liệu nào).
- ConfidenceScore: Độ tin cậy của AI (0-1). Giúp cảnh báo người dùng kiểm tra 
  lại nếu điểm quá thấp.
- PasswordHash: Mật khẩu đã mã hóa, đảm bảo an toàn nếu DB bị rò rỉ.

3. BỘ CÂU HỎI PHẢN BIỆN GIẢ ĐỊNH (PHẦN THIẾT KẾ)
--------------------------------------------------------------------------
Câu 1: Tại sao em chia làm 2 bảng Documents và FormulaEntries?
- Trả lời: Đây là mối quan hệ 1-nhiều. Một tài liệu PDF có thể chứa hàng trăm 
  công thức. Việc chia bảng giúp quản lý dữ liệu gọn gàng và truy xuất nhanh hơn.

Câu 2: "OrderIndex" trong bảng FormulaEntries dùng để làm gì?

PHẦN 8: NHẬP DỮ LIỆU TỰ ĐỘNG - SEEDING (DỰA TRÊN BLOG BÀI 8)
==========================================================================

1. KHÁI NIỆM VÀ LỢI ÍCH CỦA SEEDING
--------------------------------------------------------------------------
- Khái niệm: Seeding (gieo hạt) là quá trình tự động nạp dữ liệu mẫu vào 
  các bảng trong Database thông qua script Python.
- Lợi ích:
  + Tiết kiệm thời gian (tạo hàng nghìn bản ghi trong vài giây).
  + Kiểm thử giao diện (UI/UX) với dữ liệu thực tế (xem khung có bị vỡ không).
  + Đồng bộ dữ liệu giữa các thành viên trong nhóm phát triển.
  + Xác thực cấu trúc và các ràng buộc (Khóa ngoại, Check) của Database.

2. CÁC THÀNH PHẦN KỸ THUẬT (SQLAlchemy Session)
--------------------------------------------------------------------------
- create_engine: Trung tâm điều khiển kết nối thực tế tới PostgreSQL.
- SessionLocal: Thành phần tạo ra các phiên làm việc (Session) để truy vấn.
- commit(): Lệnh xác nhận lưu dữ liệu vào DB.
- rollback(): Hoàn tác nếu có lỗi xảy ra để tránh rác dữ liệu.

3. QUY TRÌNH THỰC HIỆN SEEDING
--------------------------------------------------------------------------
- B1: Tạo `database.py` để khởi tạo Engine và SessionLocal.
- B2: Di chuyển `models.py` vào thư mục `app/` để đồng bộ cấu trúc.
- B3: Viết script `seed.py` khởi tạo đối tượng (User, Document, Formula) 
  và dùng `db.add()`, `db.commit()` để lưu.
- B4: Chạy lệnh `python seed.py` trong Terminal.

4. GIẢI ĐÁP CÂU HỎI BÀI TẬP (BÀI 8.1 - 8.4)
--------------------------------------------------------------------------
Câu 8.1 Thư viện python-dotenv dùng để làm gì?
- Trả lời: B. Nạp các biến môi trường (Database URL) từ file .env vào hệ thống.

Câu 8.2 Đối tượng engine (create_engine) đóng vai trò gì?
- Trả lời: C. Là "trung tâm điều khiển" quản lý việc giao tiếp giữa Python và DB.

Câu 8.3 Tại sao nên dùng SessionLocal thay vì kết nối trực tiếp?
- Trả lời: B. Để SQLAlchemy quản lý các thay đổi, cho phép commit hoặc rollback.

Câu 8.4 Khi seed bảng Document có khóa ngoại UserID, ta cần làm gì?
- Trả lời: A. Phải tạo User trước, lấy ID đó gán vào Document để đảm bảo 
  tính toàn vẹn.

5. BỘ CÂU HỎI PHẢN BIỆN GIẢ ĐỊNH (PHẦN SEEDING)
--------------------------------------------------------------------------
Câu 1: "Flush" và "Commit" khác nhau thế nào trong script seeding của em?
- Trả lời: `db.flush()` đẩy dữ liệu xuống DB nhưng chưa lưu vĩnh viễn, giúp 
  em lấy được ID tự sinh để gán cho các bảng con. `db.commit()` mới chính 
  thức lưu mọi thứ vào ổ đĩa.

Câu 2: Nếu muốn tạo 1000 người dùng mẫu, em có nhập tay không?
- Trả lời: Không, em sẽ sử dụng thư viện **Faker** kết hợp với vòng lặp `for` 
  để tự động sinh ra tên, email và địa chỉ ngẫu nhiên.

Câu 3: Làm thế nào để đảm bảo dữ liệu mẫu không bị trùng lặp mỗi khi chạy?
- Trả lời: Em có thể viết thêm logic xóa sạch dữ liệu cũ trước khi nạp mới, 
  hoặc sử dụng lệnh `try-except` để bỏ qua nếu gặp lỗi trùng lặp (Unique Constraint).

1. ĐỊNH NGHĨA WEB SERVICES
--------------------------------------------------------------------------
- Là dịch vụ trao đổi dữ liệu giữa các thiết bị điện tử trên môi trường mạng 
  thông qua giao thức HTTP/HTTPS.
- Khác biệt với Trang web: 
  + Trang web: Dành cho con người đọc (HTML/CSS).
  + Web Services: Dành cho chương trình/phần mềm đọc (JSON/XML).

2. CÔNG NGHỆ FASTAPI (BACKEND)
--------------------------------------------------------------------------
- Đặc điểm: Tốc độ cao (tương đương Go/Node.js), hỗ trợ async (bất đồng bộ).
- Thành phần bổ trợ:
  + Uvicorn: Web server chuẩn ASGI để chạy ứng dụng Python.
  + Pydantic: Kiểm tra và xác thực dữ liệu (Data Validation).
- Lệnh cài đặt: `pip install "fastapi[all]"`.
- Lệnh chạy: `uvicorn main:app --reload`.

PHẦN 10: LẬP TRÌNH WEB SERVICES (DỰA TRÊN BLOG BÀI 10)
==========================================================================

1. MÔ HÌNH CLIENT-SERVER
--------------------------------------------------------------------------
- Client (Máy khách): Khởi phát yêu cầu (Request). Là thành phần chủ động.
- Server (Máy chủ): Tiếp nhận, xử lý và trả kết quả (Response). Là thành phần bị động.
- Giao thức: HTTP/HTTPS là "ngôn ngữ chung" để hai bên giao tiếp.

2. QUY TRÌNH XỬ LÝ CỦA FASTAPI
--------------------------------------------------------------------------
- Bước 1: Khởi chạy Uvicorn (Web Server).
- Bước 2: Client (Trình duyệt) gửi HTTP Request tới URL cụ thể.
- Bước 3: FastAPI kiểm tra Routing (Decorator) để tìm hàm xử lý tương ứng.
- Bước 4: Thực thi code Python và trả về kết quả (thường là JSON) cho Client.

3. CÔNG CỤ KIỂM TRA (DEVELOPER TOOLS)
--------------------------------------------------------------------------
- Phím tắt: F12 hoặc Ctrl+Shift+I.
- Tab Network: Dùng để quan sát các gói tin Request/Response, kiểm tra 
  thời gian phản hồi và dữ liệu trả về từ Server.

4. GIẢI ĐÁP CÂU HỎI BÀI 10 (10.1 - 10.8)

- Trả lời: B. Client.

- Trả lời: C. Uvicorn.

- Trả lời: C. Điều hướng yêu cầu (Routing) từ trang chủ vào hàm xử lý tương ứng.

Câu 10.7 Lệnh chạy server tự động cập nhật khi sửa code?
- Trả lời: C. uvicorn main:app --reload.
